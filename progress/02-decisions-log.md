# Decisions Log

- 2026-06-20: Polars only, no Pandas, project-wide. Rationale: DuckDB/Arrow interop and a single dataframe library reduce agent confusion about which API to reach for.
- 2026-06-20: Groq rate limits are never hardcoded as config or code constants. Rationale: free-tier RPM/RPD/TPM limits change without notice, so Groq handling must react to live 429 and `Retry-After` / `rate-limit-*` headers instead of stale assumptions.
- 2026-06-20: `dataset_columns` stores both `name` and `display_name`. Rationale: only the safe internal column name may reach SQL compilation or LLM prompts; raw CSV headers stay UI-only.
- 2026-06-20: anonymous workspaces carry explicit TTL fields via `workspaces.is_anonymous` and `workspaces.expires_at`. Rationale: demo/private throwaway workspaces need lifecycle tracking without changing ownership semantics.
- 2026-06-20: the Supabase service-role client is created only in `db/supabase_client.py` and consumed through repositories. Rationale: a single choke point reduces the chance of accidental service-role key spread across route or service modules.
- 2026-06-20: anonymous demo identity uses Supabase anonymous auth plus an auto-provisioned TTL workspace, not a parallel guest system. Rationale: anonymous users should still be real `auth.users` identities with real JWTs, shared demo-data access should come from RLS on the seeded public demo dataset, and throwaway user-owned workspaces are only for private state such as later uploads.
- 2026-06-21: anonymous demo identity uses Supabase anonymous auth + auto-provisioned TTL workspace, not a parallel guest system. Rationale: the frontend gets a real anonymous Supabase JWT directly from Auth, the backend provisions a real user-owned anonymous workspace only on first authenticated use, shared demo data stays shared via the `public_demo_read` RLS path instead of being copied per user, and expiry/cleanup remain database-driven through TTL plus the cleanup SQL function.
- 2026-06-21: fixed demo dataset random seed is `20260417`. Rationale: README screenshots and the seeded anomaly location must stay permanently reproducible across regenerations.
- 2026-06-21: the seeded demo anomaly is the week `2025-04-14` through `2025-04-20` in `West` / `Electronics`, targeted at an approximately `45%` revenue drop versus the trailing baseline. Rationale: later anomaly detection must rediscover the same statistical event without the CSV itself changing.
- 2026-06-21: Chart payload truncation policy is strictly enforced in `result_formatter.py` based on `MAX_CHART_PAYLOAD_KB`. Rationale: Centralizing it here ensures insights, anomalies, and widgets all rely on the same truncation logic without reimplementing it. Truncation sets `meta.truncated=True` and uses evenly spaced sampling for time series to avoid distorting trend lines.
- 2026-06-21: LLM prompts use internal safe column names exclusively for the emit contract; display_name is hint-only and always delimited as untrusted content. Rationale: this defends against prompt-injection via malicious CSV headers while still allowing the model to bridge the user's vocabulary to the internal schema.
- 2026-06-21: Confirmed and reinforced the no-hardcoded-limits rule for Groq rate limiting. Rationale: implemented reactive header-driven backoff (using `Retry-After` or `x-ratelimit-reset`) instead of pre-emptive constants like `sleep(2)` or `MAX_REQUESTS_PER_MINUTE`.
- 2026-06-21: Insight ranking weights are defined as constants: `WEIGHT_BUSINESS_IMPACT` (+3), `WEIGHT_HIGH_CHANGE` (+2), `WEIGHT_TOP_CATEGORY` (+2), `WEIGHT_RECENTLY_SEEN` (-1), `WEIGHT_TOO_OBVIOUS` (-1). Rationale: provides a documented baseline for easy tuning without magic numbers.
- 2026-06-22: Re-audited the Phase 3.5 chart-payload single-choke-point rule. Insight chart generation already routed through `analytics/result_formatter.py`; anomaly detectors and `/ask` had drift (`chart_payload=None` for anomalies and post-format intent metadata mutation for ask). Both are fixed so final metadata-bearing chart payloads are capped only by `result_formatter`, widgets reuse those payloads verbatim, and repositories independently reject any oversized JSONB insert/update as a non-truncating backstop.
- 2026-06-22: Anonymous workspace TTL is an access-control boundary, not just a cleanup hint. Rationale: scheduled deletion can lag, so RLS must deny reads/writes to anonymous workspaces and their child rows immediately when `expires_at <= now()`. `0005_anon_expiry_rls.sql` therefore makes `owns_workspace()` expiry-aware and updates `workspaces.workspace_owner_access` to enforce the same rule directly on the workspace table.
- 2026-06-22: `/ask` now runs a deterministic intent-resolution pass before calling Groq for common analytics phrasings and session follow-ups. Rationale: the golden suite showed too many otherwise-routine questions were failing due to avoidable LLM dependence, punctuation-sensitive follow-up parsing, and vague-but-standard business phrasing.
- 2026-06-22: Marketing page canned demo strip uses an animated loop. Rationale: Implemented the 'question→chart→summary' demo strip on the marketing page as a self-contained, pre-built/animated looping sequence (using Framer Motion) rather than a live call to the real `/ask` endpoint to avoid burning unnecessary Groq/Supabase quota on landing pages while still demonstrating UX value.
- 2026-06-22: Relative date presets in compiled DuckDB SQL are anchored to the dataset's own max date, not the server clock. Rationale: the shared demo dataset is historical, so `last_month`, `this_year`, `last_quarter`, and similar phrases must be evaluated relative to the uploaded data timeline or they silently drift into empty/incorrect windows as real time advances.
- 2026-06-22: Synthetic background `/ask` queries are used for KPI generation. Rationale: Using the existing `POST /ask` endpoint with `session_id: null` allows the UI to populate canonical dashboard metrics (revenue, AOV) via the robust query compilation engine without polluting the user's conversational history or polluting the UI with automated background artifacts.
- 2026-06-22: Frontend-only staged loading. Rationale: the staged loading sequence ("Reading schema...", "Building query plan...", "Rendering chart...") during a slow NLQ query is a purely deterministic, timed frontend illusion designed to improve perceived progress. The backend does not actually stream intermediate execution states, and no streaming endpoint should be implemented.
- 2026-06-22: `apps/api/requirements.txt` is finalized with exact pinned versions instead of floating ranges. Rationale: this codebase now depends on a live-verified integration stack across FastAPI, Supabase, DuckDB, Polars, scikit-learn, and the golden suite, so reproducible install parity matters more than opportunistic upgrades. The final manifest also removes dead scaffold dependencies (`groq`, `pytest-asyncio`) and keeps only packages the code or tests directly import or require at runtime.
- 2026-06-22: deployment verification is Render-first and Render-only for now; Hugging Face Spaces is not maintained as an actively verified fallback artifact. Rationale: Docker is not available in this workspace for a trustworthy local image build, while the actual deploy requirement is a fresh-environment `uvicorn` boot with documented env vars. Shipping an unverified Dockerfile would add risk without increasing confidence.
- 2026-06-22: web app initialized with npm as package manager, Geist as the default font stack, and the Ultramarine accent (--accent #2A2FED). Rationale: npm aligns with the current workspace setup, Geist satisfies the strict tabular-nums and metric-style requirements natively, and Ultramarine establishes the stated visual identity early to prevent random color drifts.
- 2026-06-22: Follow-up question match cue is purely cosmetic. Rationale: The frontend heuristically highlights follow-up intents (like "break that down by" or "compare") visually with a "Follow-up" badge, but the *actual* resolution and merging of context is entirely a backend Phase 5 responsibility. The frontend must never intercept, alter, or "smart merge" the query payload.
